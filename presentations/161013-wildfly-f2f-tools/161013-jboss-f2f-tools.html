mvnw, a.k.a. Maven Wrapper

  • Idea borrowed from Gradle’s gradlew

  • mvnw - a shell script that

    • downloads the Maven version configured in .mvn/wrapper/maven-wrapper.properties
    • forwards the CLI args to that version’s mvn

  • Instead of usual mvn clean install, users are supposed to use ./mvnw clean install
  • mvnw.cmd for windows

Why:

  • A standard way of doing (part of) what we do in our build.sh

  • Maven installations under ~/.m2/wrapper/ shareable among multiple projects

    • NB: build.sh installs to ./tools

https://github.com/takari/maven-wrapper#maven-wrapper

.mvn/maven.config, .mvn/jvm.config (Maven 3.3.1+)

.mvn/maven.config - a new home for MAVEN_OPTS

  • Default goals, e.g. clean package

  • Other Maven options to have active without having to specify them on CLI every time

    • e.g. --fail-at-end --batch-mode --settings ./my-setting.xml

.mvn/jvm.config

  • The same thing for JVM args, like -Xmx2048m -Xms1024m

Takari Concurrent Safe Local Repository

Default Maven local repository implementation not designed for

  • Multiple instances of Maven on the same machine
  • Multiple threads of one Maven process

Concurrent access may result in corrupting the local repository

Takari Concurrent Safe Local Repository - a core extension for Maven 3.3.1+ https://github.com/takari/takari-local-repository

Takari Lifecycle - incremental builds (non-proposal)

mvn package without clean

  • Not truly incremental
  • Not reliable

Takari Lifecycle

  • Own set of core plugins (clean, compile, package, …​)
  • All of them configured through takari-lifecycle (a wrapper plugin)
  • takari-lifecycle overrides the default Maven lifecycle mapping

Concerns:

  • Big migration overhead, required configuration options missing
  • Based on Eclipse Compiler, which behaves differently in some situations

srcdeps - shorten the integration cycles

Figure 1. WildFly Components dependencies, strongly simplified

Change’s travel through the component hierarchy

Given that Component A depends on Component B

  1. Merge a change (Fix, Feature, …​) in B
  2. Wait till B gets released
  3. Upgrade B in A
  4. Optionally adapt A
  5. Test B inside A
  6. Bug in B uncovered by A’s integration tests? - Goto 1
  7. Integration in A successful? - wait for release of A

What is wrong with the above process?

  • Infrequent releases of components
  • Releases may collect multiple changes → regression causes are harder to find

How can we propagate smaller change sets faster?

With the stock Maven, there are only two options to shorten the cycles:

  • Use SNAPSHOTs
  • Relase often

Remote SNAPSHOTs are evil

You never know what you get

  • local/remote SNAPSHOT?
  • latest today != latest tomorrow

The build of a component depending on another SNAPSHOT component is not reproducible

Best practice: Remote SNAPSHOTs always off (esp. for CI)

Release components often

  • 100% reproducible
  • Releasing may cost developers' time (doable by CI)
  • Many short living dev releases clutter maven repositories

  • The component’s team must want to release

    • Governed outside of one’s own organization?

Let’s forget the stock Maven for a moment…​

  • The build of every git commit deterministic enough
  • Why don’t we depend on git commits directly?
  • We could build the components on the fly from their sources!

Welcome srcdeps!

  • No need for releasing and even having a public maven repo (not our case)
  • Nevertheless we can fully get rid of releases done just for the sake of testing and integration

Change’s travel with srcdeps

Parts in bold eliminated

  1. Merge a change (Fix, Feature, …​) in B
  2. Wait till B gets released
  3. Upgrade B in A
  4. Optionally adapt A
  5. Test B inside A
  6. Bug in B uncovered by A’s integration tests? - Goto 1
  7. Integration in A successful? wait for release of A

How srcdeps work (1/2)

  1. A custom implementation of Local Maven Repository (via Maven Core Extensions mechanism)

  2. A convention for version strings to express the commit ID to build

    <dependency>
  <groupId>org.wildfly.core</groupId>
  <artifactId>wildfly-core-featurepack</artifactId>
  <version>3.0.0-SRC-revision-deadbeef</version>
</dependency>
  3. Provide a configuration that maps groupIds to source repository URLs

How srcdeps work (2/2)

When an artifact with *-SRC-revision-{commitId} version is looked up in Maven local repo

  • Find a git URL for it
  • Checkout the source
  • Change the versions in the pom.xml files to whatever *-SRC-revision-{commitId} was requested
  • Build and install locally

.mvn/extensions.xml

A place to declare Core Extensions of Maven 3.3.1+

<extensions>
  <extension>
    <groupId>org.l2x6.srcdeps</groupId>
    <artifactId>srcdeps-maven-local-repository</artifactId>
    <version>1.0.1</version>
  </extension>
</extensions>

.mvn/srcdeps.yaml - the srcdeps configuration file.

A minimal srcdeps.yaml file

# `configModelVersion` exists just in case we'll have to change the schema of srcdeps.yaml in the future.
# Required, no default, only 1.0 supported at the moment
configModelVersion: 1.0

# `repositories` - a map of SCM repositories.
# Required, no default
repositories:

  # the `id` of the source repository, typically the `groupId` of its artifacts,
  # or any other string unique at this level of the yaml file.
  org.l2x6.maven.srcdeps.itest:

    # `selectors` - a way how to map dependency artifacts to source repositories.
    # At the moment, the association is given by the exact string match between the `groupId` of the dependency
    # artifact and one of the selectors listed here.
    #
    # At least one selector is required
    selectors:
    - org.l2x6.maven.srcdeps.itest # This SCM repository entry will be used for building artifacts
                                   # with groupId org.l2x6.maven.srcdeps.itest

    # `urls` - a list of URLs to checkout the sources of the given dependency. If you list multiple SCM
    # repos here then only the first successful checkout will count.
    #
    # At least one URL is required
    urls:
    - git:https://github.com/l2x6/srcdeps-test-artifact.git

Miscelaneous srcdeps features

  • Depend on branches and tags in addition to revisions
  • Configurable verbosity of dependency builds
  • Custom arguments for dependency builds (goals, profiles, …​)
  • -DskipTests by default
  • IO redirects
  • Should (untested!) work for vitually any kind of a dependency incl. parent, managed imports and even plugins

Limitations

Can you think of any?

  • Only Maven and git supported ATM (contributions welcome!)
  • Tools not acquainted with srcdeps (IDEs, static pom.xml analysers, …​) will see the -SRC- deps as non-available

  • However immutable git commits are, they can still disappear from repos, or even the whole repo can be deleted

    • Best practice: use srcdeps only against your own repos or your own mirrors of third party repos

  • There is still a few things that may differ among devs: java version, mvn version

    • Projects should use mvnw with a fixed mvn version
    • Enforcer plugin for Java version

  • The -SRC- artifacts (jars, wars, …​) built by you and me will typically not be binary equal

    • Typically will make no harm
    • Where would this matter actually?

Things to remember about srcdeps

  • Shortens the integration/delivery cycles in complex dependency hierarchies
  • May save some time of a release engineer
  • May eliminate the need for a public Maven repository
  • No action or consent from the dependency project required
